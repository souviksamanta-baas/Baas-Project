---
title: "Authentication"
description: "How API clients authenticate with Nexolia endpoints."
---

Nexolia uses different authentication mechanisms depending on the endpoint type.

## Owner API authentication

Owner-facing endpoints use Supabase Auth access tokens.

Send the token in the standard `Authorization` header:

```http
Authorization: Bearer <supabase-access-token>
```

Current owner-authenticated endpoints include:

- `POST /organizations/invites` — create a staff invite (owner bearer token)
- `POST /organizations/invites/accept` — accept invite after phone verification
- `POST /whatsapp/connection/register` — register merchant WhatsApp Cloud API number
- `POST /ai/drafts/{draftId}/approve`
- `POST /ai/drafts/{draftId}/reject`
- `POST /ai/copilot/query`

These endpoints validate the token server-side before allowing access to organization-scoped data.

## Public auth OTP endpoints

WhatsApp login OTP uses the Nexolia **platform WABA** (not merchant inbox numbers). These routes are unauthenticated and rate-limited at the edge in production:

- `POST /auth/otp/whatsapp/request` — send Meta authentication-template OTP
- `POST /auth/otp/whatsapp/verify` — verify OTP and return a Supabase `tokenHash`

The mobile client exchanges `tokenHash` with Supabase Auth (`verifyOtp`) to establish the session.

**Production pilot:** owner login ships **email-only** (`EXPO_PUBLIC_AUTH_LOGIN_CHANNELS=email`). WhatsApp OTP remains implemented but disabled until Meta business verification and a Spanish AUTHENTICATION template are approved.

Email and optional SMS OTP use Supabase Auth directly from the mobile client (no NestJS route).

## Scheduler authentication

Operational maintenance jobs use a shared scheduler secret.

Send the secret in:

```http
x-baas-job-secret: <configured-secret>
```

Current scheduler endpoint:

```http
POST /tasks/run-maintenance
```

The API expects the value to match `BAAS_TASKS_JOB_SECRET`. If the secret is missing from the API environment, the endpoint returns `503 Service Unavailable`.

## WhatsApp webhook signature

WhatsApp inbound webhooks use the Meta signature header:

```http
x-hub-signature-256: sha256=<signature>
```

When `WHATSAPP_APP_SECRET` is configured, the API validates the raw request body against this HMAC signature before processing the payload.

## Security guidance

- Do not expose service-role Supabase keys to clients.
- Do not send `x-baas-job-secret` from mobile or browser clients.
- Rotate scheduler and webhook secrets if they are accidentally exposed.
- Treat webhook signatures as required in production.
